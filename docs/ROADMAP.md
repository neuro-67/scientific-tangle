## Definition of Done (MVP-демо)
Жюри в браузере вводит вопрос из ТЗ (напр. про обессоливание воды или циркуляцию католита) и получает:
- Структурированный ответ с **цитатами и уровнем достоверности**.
- **Подграф** знаний (материал→процесс→оборудование→результат).
- Фильтры: **RU/foreign**, год, числовой диапазон.
- Подсветку **пробела/противоречия** хотя бы на одном примере.
- Хотя бы один **дашборд** покрытия знаний.

Всё это на реальном корпусе (предоставленный PDF + каталог экспериментов).

- [x] Extraction через LLM (RouterAI, `qwen/qwen3-30b-a3b-instruct-2507`) по JSON-схеме — прогнано на реальном корпусе (4 документа, сотни чанков), `response_format` (JSON Schema, server-side constrained decoding) + `max_tokens=6000` дают 96% успешных чанков и 0 нарушений онтологии.
- [x] Normalize: **term_aliases реализован** (`nlp/term_aliases.json` + `apply_term_aliases()` в `nlp/run_corpus_test.py`) — проверено на синтетическом тексте: "ПВП"/"печь взвешенной плавки"/"fluidized bed furnace" схлопываются в один узел. `pint`-нормализация единиц — **не реализована**. Write в Neo4j с провенансом (`Source`/`has_source`) — сделано и проверено на реальном графе (359 nodes / 469 edges). Индексы/констрейнты — есть (`nlp/ingestion/neo4j_import.py`).
- [x] **Механическая верификация чисел** (не было в исходном плане, но необходимо): `verify_measurements()` сверяет каждое число из Measurement с сырым текстом документа — поймала реальную галлюцинацию модели (4 придуманных значения, включая 101325 Па = атмосферное давление) на кейсе Луканиной М.А.

### Query (ML-2)
- [x] QuerySpec-парсер (RouterAI `gemini-3.1-flash-lite`) — проверен на золотых вопросах из ТЗ и на реальном compare-запросе против живого графа.
- [x] Гибрид: Qdrant + Cypher по QuerySpec + RRF — **прогнан end-to-end против реального Neo4j+Qdrant**. По пути найдены и исправлены 4 бага, из-за которых поиск молча возвращал бы пусто/падал: (1) Cypher искал `n.canonical_name`, в графе — `n.id`; (2) Cypher использовал типы связей строчными буквами, в базе они `UPPER_SNAKE`; (3) Qdrant payload использовал разные имена полей на записи/чтении; (4) `qdrant_client.search()` не существует в установленной версии клиента (1.18.0), нужен `query_points()`. Также RRF-merge терял все метаданные (title/year/geography/confidence/span/extracted_at) перед синтезом — исправлено.
- [x] Synthesis-промпт → структурированный ответ с цитатами, **датой актуализации** (`extracted_at`, отдельно от `year` источника) и **рекомендациями** (`recommendations`: similar_case/related_topic, per case-specification.md). Подграф (`get_subgraph`) — проверен на реальных данных (38 строк по 2 хопам от "печь Ванюкова"), даёт связную цепочку материал→процесс→оборудование→результат.
- [x] Межисточниковые противоречия (`disagreements`) — механизм в промпте синтеза (LLM сравнивает факты из РАЗНЫХ источников среди retrieved-кандидатов); проверено на синтетическом конфликте (два источника с разным % извлечения) — заполняется корректно; на реальных 4 документах корпуса естественных противоречий между источниками не нашлось (материал у них разный), так что `disagreements: []` в реальном прогоне — ожидаемо пусто, а не баг.
- [ ] **Известный остаточный риск**: Cypher graph-search матчит по точному `id`, поэтому при перефразировке QuerySpec-парсером (например "плавка в печи Ванюкова" вместо канонического имени узла) graph-ветка может вернуть 0 строк и всю выдачу тянет только Qdrant vector search — семантически ещё работает, но это слабое место без fuzzy/embedding-резолюции сущностей на входе запроса.
- [x] **`backend/src/app/features/query/` (реальный HTTP-путь `/query/ask`) проверен end-to-end** — это отдельная от `nlp/query/retrieval/*` реализация (свой `Neo4jGraphSearch`, свой `QdrantVectorSearch`), не разделяет код с тем, что было протестировано раньше. Дозаписал туда: `HAS_SOURCE`/`span`/`extracted_at` (раньше не запрашивались вообще), ranking по `confidence` через `CASE` (было `ORDER BY` по вперемешку строкам/числам — не работало бы), тот же payload-фикс `entity_id`/`doc_id`, что и в `nlp/`.
- [x] **Cross-encoder реранк вместо RRF** (см. P2 ниже) — `AskQuestionHandler._rrf_merge` был сломан (считал RRF-скор и тут же его выбрасывал, возвращая только graph-результаты, vector-результаты полностью терялись). Заменён на конкатенацию+дедуп, реальное ранжирование — кросс-энкодер `BAAI/bge-reranker-v2-m3` (модель реально грузится в этом окружении, ~30с, без сегфолтов).
- [x] **Найден и исправлен краш-баг в synthesis**: LLM иногда пишет `"не указан"`/`"неизвестна"` вместо JSON `null` в `year`/`geography`, когда данных нет — раньше это валило Pydantic-валидацию целиком и весь синтезированный ответ выбрасывался в пользу грубого fallback. Добавлен `field_validator` (мягкое приведение к `null`) + инструкция в промпт.
- [x] **Числовые диапазоны/условия** (ТЗ: "концентрации сульфатов <200 мг/л") — были сломаны, теперь работают, нашёл и исправил 2 независимых бага: (1) модель иногда пишет буквальный `<`/`>` вместо разрешённых `<=`/`>=`/`=`, схема падала с `ValidationError` и весь numeric_constraint терялся через откат на rule-based fallback (тот не умеет извлекать числа вообще) — добавлено мягкое приведение `<`→`<=`/`>`→`>=`; (2) Cypher-фильтр в ОБЕИХ реализациях (`nlp/` и backend) сравнивал имя свойства с `meas.id`/`meas.property`, а название свойства реально лежит на отдельном узле `Property` через `MEASURES_PROPERTY` — фильтр никогда ни с чем не совпадал. Проверено синтетическим тестом (150 мг/л и 250 мг/л при `<=200`): до фикса — 0 результатов из ожидаемого 1, после — корректно 1/1.
- [x]⚠️ **Сравнительные запросы** (РУ vs мировая практика, вариант А vs Б) — `intent=compare` и поле `compare` парсятся верно, но `compare` **нигде не используется дальше** (нет отдельного Cypher/Qdrant-фильтра, нет двухпроходного retrieval) — вся "сравнительность" ответа держится на том, что LLM в synthesis сама сопоставляет факты по geography-тегам в контексте. Работает, когда retrieval нашёл факты по обеим сторонам сравнения; проверено на реальном вопросе про РУ/зарубежные стандарты (в графе реально есть 21 таких стандарта) — retrieval их не нашёл (сработал уже известный риск exact-id матчинга выше), и система честно ответила "нет данных" вместо выдумывания — безопасно, но не то же самое, что "сравнение работает".

## Критический путь и риски

| Риск | Митигирование |
|---|---|
| compose/инфра не поднимается вовремя | Фаза 0 — приоритет №1, все включаются; fallback — локальный запуск сервисов |
| Извлечение из PDF шумное (сканы/таблицы) | Начать с текстового слоя; таблицы — отдельными чанками; ручной отбор 20–30 «чистых» страниц для демо |
| Качество извлечения чисел | `evidence`-цитаты + `pint`; для демо — проверенное подмножество фактов |
| Не сходится сквозной срез | Gate дня 2: режем фичи, сохраняем 1 рабочий сценарий любой ценой |
| Стоимость/лимиты LLM API | быстрая модель для потока, качественная модель точечно; кэш ответов; ограничить объём корпуса демо |
| Синхронизация контрактов ML↔Backend | Схемы зафиксированы в Фазе 0, любые правки — через общий файл схем |
