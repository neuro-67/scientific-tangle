## Definition of Done (MVP-демо)
Жюри в браузере вводит вопрос из ТЗ (напр. про обессоливание воды или циркуляцию католита) и получает:
- Структурированный ответ с **цитатами и уровнем достоверности**.
- **Подграф** знаний (материал→процесс→оборудование→результат).
- Фильтры: **RU/foreign**, год, числовой диапазон.
- Подсветку **пробела/противоречия** хотя бы на одном примере.
- Хотя бы один **дашборд** покрытия знаний.

Всё это на реальном корпусе (предоставленный PDF + каталог экспериментов).

## Роли
- **FS-1** — Backend core: FastAPI-скелет, auth/RBAC, Postgres, оркестратор запроса, аудит, экспорт.
- **FS-2** — Frontend: React/TS, поиск, ответ+цитаты, граф (Cytoscape), дашборд.
- **ML-1** — Ingestion: parse→chunk→extract→normalize→write в Neo4j.
- **ML-2** — Query/Retrieval: QuerySpec-парсинг, гибридный поиск, синтез-промпт, Qdrant.

## Фаза 1 — Вертикальный срез (день 1, вторая половина → утро дня 2)
Цель: сквозной путь на 1–2 документах и 3–5 вопросах. Идём вертикально, не вширь.

### Backend (FS-1)
- [ ] Модели Postgres: `users, roles, documents, jobs, audit_log, term_aliases` + Alembic.
- [ ] Auth: JWT + argon2, сид-пользователи по 5 ролям.
- [ ] `POST /documents` (upload → MinIO → job в arq), `GET /documents/{id}` статус.
- [ ] `POST /query` — оркестратор: вызывает retrieval (ML-2) → synthesis → ответ+аудит.

### Ingestion (ML-1)
- [ ] Парсинг предоставленного PDF (PyMuPDF) + каталога экспериментов (CSV/справочники).
- [ ] Chunk + language detect.
- [ ] Extraction через LLM по JSON-схеме на 5–10 чанках (сначала качество, потом объём).
- [ ] Normalize (term_aliases + pint) → Write в Neo4j с провенансом. Индексы/констрейнты.

### Query (ML-2)
- [ ] QuerySpec-парсер (LLM) на 3–5 эталонных вопросах из ТЗ.
- [ ] Гибрид: Qdrant top-k + Cypher по QuerySpec + числовой фильтр. RRF-объединение.
- [ ] Synthesis-промпт → структурированный ответ с цитатами. Вернуть подграф.

### Frontend (FS-2)
- [x] Экран поиска: строка запроса + многоуровневая фильтрация (материал/процесс/гео/год/уровень достоверности). _(на мок-апи MSW)_
- [x] Экран ответа: текст + карточки источников (год, гео, достоверность, span). _(на мок-апи MSW)_
- [x] Логин, хранение JWT. _(на мок-апи MSW)_

**Gate дня 2 (утро):** один вопрос из ТЗ проходит сквозь всю систему на реальном корпусе и возвращает ответ с цитатами в UI. **Это самый важный чекпоинт — если срез не сходится к утру дня 2, режем фичи, а не сдвигаем срез.**

## Критический путь и риски

| Риск | Митигирование |
|---|---|
| compose/инфра не поднимается вовремя | Фаза 0 — приоритет №1, все включаются; fallback — локальный запуск сервисов |
| Извлечение из PDF шумное (сканы/таблицы) | Начать с текстового слоя; таблицы — отдельными чанками; ручной отбор 20–30 «чистых» страниц для демо |
| Качество извлечения чисел | `evidence`-цитаты + `pint`; для демо — проверенное подмножество фактов |
| Не сходится сквозной срез | Gate дня 2: режем фичи, сохраняем 1 рабочий сценарий любой ценой |
| Стоимость/лимиты LLM API | быстрая модель для потока, качественная модель точечно; кэш ответов; ограничить объём корпуса демо |
| Синхронизация контрактов ML↔Backend | Схемы зафиксированы в Фазе 0, любые правки — через общий файл схем |

## Анти-цели (не делаем на хакатоне)
- Полный OWL-ризонер и SHACL-валидацию — только документируем.
- Дообучение ruBERT/spaCy NER — только в roadmap доклада.
- SSO/LDAP, шифрование полей, PDF-рендер отчётов — P2/roadmap.
- Масштаб на 1 млн сущностей вживую — показываем архитектурную готовность (индексы), но грузим демо-объём.
