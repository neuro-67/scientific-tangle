# Как получить одну и ту же графовую БД у всей команды

## Проблема

`docker compose up` у каждого локально поднимает СВОЙ Neo4j/Qdrant с пустыми
данными. Если ML-1 прогнал ingestion у себя на машине, у ML-2/FS-1/FS-2 при
локальном запуске будет пустой граф — retrieval и `/query/ask` не найдут
ничего, хотя код рабочий.

## Вариант A (рекомендуется): один общий инстанс на VPS

Решает сразу 2 задачи: общая БД для команды + обязательный пункт сдачи
"ссылка на развёрнутое решение".

1. Любой VPS с Docker (от 4GB RAM).
2. `git clone` репозитория, `cp .env.example .env`, вписать реальные секреты
   (`LLM_API_KEY`, `ROUTERAI_API_KEY`, `JWT_SECRET`, пароли БД).
3. `docker compose up -d --build`.
4. **Не прогоняйте LLM-экстракцию заново на VPS** — она уже сделана и
   провалидирована локально (см. ниже, как перелить готовые графы).
5. Наружу открывайте только `backend` (8000) и `frontend` (5173/443).
   `Neo4j`/`Qdrant`/`Postgres` держите внутри docker-сети — Qdrant по
   умолчанию без аутентификации, Neo4j-порт наружу — лишняя площадь атаки.
   Если нужен прямой доступ к БД для скриптов — через SSH-туннель на VPS,
   не публичный порт.
6. Все (команда, потом жюри) работают через один backend URL вместо
   `localhost`.

## Вариант B: перелить уже готовые графы в свой/общий Neo4j+Qdrant

Не нужно ничего пересчитывать через LLM — экстракция стоит времени и денег
(RouterAI-вызовы), а результат уже есть в виде JSON-файлов.

```bash
# 1. Скажите Neo4j-import/Qdrant-upload, КУДА писать -- через env, а не через
#    правку кода. По умолчанию оба смотрят на localhost:
export NEO4J_URI=bolt://<адрес-инстанса>:7687      # или localhost при локальном тесте
export NEO4J_USER=neo4j
export NEO4J_PASSWORD=<пароль>
export QDRANT_HOST=<адрес-инстанса>
export QDRANT_PORT=6333

# 2. Импортируете уже готовые, провалидированные графы -- без единого LLM-вызова:
python -m nlp.ingestion.neo4j_import nlp/corpus_test_results/clean/*.json
python -m nlp.ingestion.qdrant_upload nlp/corpus_test_results/clean/*.json
```

`MERGE`-запись в `neo4j_import.py` идемпотентна по `id` — повторный импорт
того же файла на тот же инстанс не создаёт дублей, можно гонять сколько
угодно раз без опасений.

**Важно про приватность:** `nlp/corpus_test_results/clean/*.json` — намеренно
НЕ в git (в `.gitignore`), потому что реальные документы кейса иногда
содержат личные email авторов (LLM извлекает их в узлы `Expert`). Делитесь
этими файлами с командой напрямую (не через публичный репозиторий), либо
сотрите `properties.email` перед тем, как класть куда-то ещё, кроме
приватного канала команды.

## Вариант C: офлайн-дамп (если нет сети между машинами вообще)

```cypher
CALL apoc.export.cypher.all("export.cypher", {})
```

APOC уже включён в `docker-compose.yml`. Файл `export.cypher` передаёте
любым способом (флешка, мессенджер), коллеги прогоняют `cypher-shell <
export.cypher` у себя. Для Qdrant отдельно — snapshot API
(`POST /collections/{name}/snapshots`); учтите, что это ДВА раздельных
экспорта, узлы графа и векторы связаны по `id`, версии могут разъехаться,
если экспортировать не одновременно.

## Не гоняйте ingestion параллельно на одних и тех же документах

`MERGE`-идемпотентность защищает от дублей узлов, но каждый повторный
LLM-прогон того же документа — это впустую потраченное время/деньги на
реэкстракцию. Пусть ingestion новых документов делает один человек в общий
инстанс, остальные — читают.
