# Архитектура системы «Научный клубок»

> Единая карта знаний R&D для горно-металлургической отрасли.
> GraphRAG-система: NLP-извлечение → граф знаний → гибридный поиск → синтез ответов с цитированием.

---

## 1. Ключевая идея архитектуры

Система строится вокруг паттерна **GraphRAG** (Graph + Retrieval-Augmented Generation):

1. **Извлечение (offline)** — документы разбираются NLP-пайплайном на сущности, связи и числовые ограничения по онтологии предметной области. Всё пишется в **граф знаний** (Neo4j) с провенансом (источник, достоверность, дата) и в **векторное хранилище** (Qdrant) для семантического поиска.
2. **Ответ (online)** — запрос на естественном языке парсится в структурированные фильтры (материал / процесс / география / диапазоны / время), выполняется **гибридный поиск** (вектор + обход графа + числовые фильтры), а LLM синтезирует структурированный ответ с цитатами, уровнем консенсуса и указанием пробелов.

Почему не «чистый RAG по чанкам»: ТЗ требует многопараметрических запросов с обходом 3–4 уровней связей, числовых диапазонов, различения отечественной/зарубежной практики и обнаружения противоречий/пробелов. Это структурные операции — их естественно выражать в графе, а не в косинусной близости.

## 3. Сервисы и зоны ответственности

| Сервис | Технология | Ответственность | Команда |
|---|---|---|---|
| **Frontend** | React + TS + Vite | Поиск, отображение ответа с цитатами, визуализация графа, дашборды, экспорт | Fullstack |
| **Backend API** | FastAPI (Python 3.11) | REST, аутентификация/RBAC, оркестрация запроса, аудит, экспорт, вебхуки | Fullstack |
| **NLP/Ingestion Worker** | Python + arq (Redis) | Парсинг документов, извлечение сущностей/связей/чисел, нормализация, резолвинг, запись в граф | ML |
| **Graph DB** | Neo4j 5 | Хранение графа знаний, многохоповые Cypher-запросы, провенанс, версионирование фактов | ML + Backend |
| **Vector store** | Qdrant | Семантический поиск по чанкам и сущностям (эмбеддинги RU/EN) | ML |
| **Relational DB** | PostgreSQL 16 | Пользователи, роли, аудит, статусы задач, метаданные документов, подписки | Backend |
| **Cache/Queue** | Redis | Очередь ingestion-задач, кэш ответов и эмбеддингов | Backend |
| **Object store** | MinIO (S3 API) | Оригиналы документов (PDF/DOCX) | Backend |
| **LLM** | Claude API (`claude-opus-4-8` / `claude-sonnet-4-6`) | Структурированное извлечение, парсинг запроса, синтез ответов | ML |

> Всё поднимается одним `docker-compose` (см. `infra/`). Для хакатона MinIO можно заменить локальной ФС, а Qdrant запускать в embedded-режиме — но compose проще и ближе к «развёрнутому решению» для жюри.

## 4. Поток данных: Ingestion (offline)

```
Upload → ObjectStore + Postgres(Document, status=queued) → enqueue(Redis)
   │
   ▼  [NLP Worker]
1. Parse       PDF/DOCX → текст + таблицы + layout   (PyMuPDF, unstructured)
2. Language    ru/en детект по чанкам                 (fasttext-langdetect)
3. Chunk       семантические чанки ~800 токенов, overlap
4. Extract     LLM structured output по онтологии:
                 entities[], relations[], measurements[]  (Claude, JSON-schema)
5. Normalize   канонизация терминов и единиц:
                 «electrowinning»→«электроэкстракция», «мг/дм³»→«мг/л»
6. Resolve     entity resolution: сопоставление с существующими
                 узлами графа (embedding + fuzzy + правила)
7. Embed       эмбеддинги чанков и сущностей → Qdrant
8. Write       MERGE узлов/связей в Neo4j с провенансом:
                 (:Source), confidence, extracted_at, doc_id, span
9. Status      Postgres(Document, status=done) + метрики
```

**Идемпотентность:** повторная загрузка того же документа не дублирует узлы (детерминированные ключи + `MERGE`). Ошибка на любом шаге → задача помечается `failed` с трассировкой, документ не остаётся в полуобработанном состоянии (частичная запись откатывается на уровне doc-транзакции провенанса).

## 5. Поток данных: Query / Answer (online)

```
NL-запрос
   │
   ▼  [Backend Orchestrator]
1. Query Parse   LLM structured output → QuerySpec:
                   materials[], processes[], geo(RU/foreign),
                   time_range, numeric_constraints[], intent(search|review|compare|gap)
2. Retrieve (параллельно):
      a) Vector    Qdrant top-k чанков/сущностей по эмбеддингу запроса
      b) Graph     Cypher по QuerySpec: обход uses_material / operates_at_condition /
                   produces_output / described_in (3–4 хопа) + фильтры
      c) Numeric   фильтр Measurement-узлов по диапазонам (≤300 мг/л и т.п.)
3. Merge+Rerank  объединение кандидатов, дедуп, реранк (cross-encoder / RRF)
4. Assemble      контекст: факты графа + чанки-подтверждения + провенанс
5. Synthesize    LLM → структурированный ответ:
                   consensus[], disagreements[], per-source citations,
                   confidence, gaps, related experts/labs
6. Subgraph      выборка узлов/связей для визуализации на фронте
7. Audit         лог запроса (Postgres) + кэш (Redis)
   │
   ▼
Ответ = { answer, sources[], subgraph, gaps, experts[], confidence }
```

Целевой SLA (ТЗ): 3–5 c на сложный запрос до 1 млн сущностей. Достигается за счёт индексов Neo4j (по типу/канон. имени/гео/году), HNSW в Qdrant и кэша по нормализованному QuerySpec.

## 6. Модель верификации знаний (провенанс)

Каждый факт (связь) в графе несёт провенанс — это прямое требование ТЗ:

- **Источник** — связь `(:Fact)-[:described_in]->(:Publication|Report|Patent)` + `span` (страница/офсет).
- **Достоверность** — `confidence ∈ {high, medium, low}`: `high` — внутренний верифицированный отчёт/патент, `medium` — рецензируемая публикация, `low` — авто-извлечение без подтверждения.
- **Дата актуализации** — `extracted_at`, `valid_from`, `valid_to`.
- **Верификация экспертом** — `validated_by` (эксперт + дата) повышает достоверность.
- **Противоречия** — связь `contradicts` между конфликтующими фактами; подсвечивается в UI.

**Версионирование:** факты не перезаписываются, а закрываются (`valid_to`) и создаётся новая версия. История выводов сохраняется.

Подробная онтология — см. [ONTOLOGY.md](./ONTOLOGY.md).

## 7. Многоязычность и синонимы

- **Извлечение** — Claude обрабатывает RU/EN нативно, извлечение идёт в единую канон. форму.
- **Controlled vocabulary** — таблица `term_aliases` (Postgres) + справочники из ТЗ: `электроэкстракция ↔ electrowinning`, `ПВП ↔ печь взвешенной плавки ↔ fluidized bed furnace`. Канон. имя — ключ узла в графе.
- **Эмбеддинги** — мультиязычная модель (`intfloat/multilingual-e5-large` или `BAAI/bge-m3`), поэтому запрос на русском находит англоязычные чанки.

---

## 8. Безопасность и доступ (RBAC)

- **Роли:** `researcher`, `analyst`, `project_lead`, `admin`, `external_partner`.
- **Разграничение данных:** каждый `Document`/`Fact` имеет `sensitivity ∈ {public, internal, confidential}`. Внешний партнёр не видит `confidential`; фильтр применяется как на этапе retrieval (Cypher/Qdrant filter), так и на выдаче.
- **Аудит:** каждый запрос/просмотр/экспорт → `audit_log` (кто, что, когда, фильтры). Требование ТЗ и основа дашбордов активности.
- **Auth:** JWT (access+refresh), пароли — argon2. Для хакатона — простая регистрация + сид-пользователи по ролям.

## 9. Аналитика и синтез (что отдаёт система сверх «списка ссылок»)

- **Литературный обзор** — группировка источников по методу/году/географии, консенсус vs разногласия, число подтверждающих источников.
- **Пробелы в знаниях** — Cypher находит незаполненные комбинации `Material × Process × Condition` (нет `Experiment`), а также темы, описанные только в RU **или** только в foreign литературе.
- **Рекомендации** — похожие кейсы из смежных доменов (через общие процессы/материалы), эксперты/лаборатории по теме.
- **Сравнение технологий** — таблица по параметрам (эффективность, CAPEX, применимость в холодном климате, эко-ограничения) из `Property`/`Measurement`.

---

## 10. Масштабируемость и расширяемость

- **Новые домены** (гидромет / пиромет / экология / переработка отходов) — добавляются как теги-таксономия и подтипы `Process`, без изменения схемы.
- **Новые типы сущностей** (напр. `EconomicIndicator`, данные с датчиков) — онтология в OWL/SHACL-стиле, extraction-схема параметризована; добавление типа = новый JSON-schema блок + Cypher-констрейнт.
- **Новые источники** — ingestion-воркеры горизонтально масштабируются (stateless, общая очередь Redis).
- **FAIR** — стабильные URI узлов, экспорт JSON-LD, метаданные и лицензии → Findable/Accessible/Interoperable/Reusable.

---

## 11. Наблюдаемость и надёжность

- **Логирование** — structlog (JSON), корреляция по `request_id` через все сервисы.
- **Метрики** — Prometheus-эндпоинт FastAPI (latency, ошибки extraction, длина очереди). Опционально Grafana.
- **Обработка ошибок ingestion** — dead-letter в Redis, статус `failed` в Postgres с причиной, ретраи с backoff.
- **Health-чеки** — `/health` для каждого сервиса; готовность зависимостей (Neo4j/Qdrant/PG) при старте.
