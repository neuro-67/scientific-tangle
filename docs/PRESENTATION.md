# Материалы для презентации «Научный клубок»

## 1. Проблема (контекст, для слайда/речи)

В горно-металлургическом R&D знания не теряются — они рассеиваются. Институт годами накапливает отчёты об экспериментах, статьи, доклады, справочники, а решение конкретной задачи требует не «найти документ», а **сопоставить факты из десятков источников**: какие концентрации сульфатов уже пробовали при обессоливании воды, какая скорость циркуляции католита считается оптимальной в мировой практике, что показали эксперименты по распределению драгметаллов между штейном и шлаком за последние 5 лет. Ручной сбор такой информации — это недели работы, дублирование уже сделанных литературных обзоров, и риск конфликтующих выводов, если у организации нет единой верифицированной базы.

Из ТЗ: «Ответ на вопрос "какие технические решения подачи электролита в ванны электроэкстракции никеля существуют в мировой практике?" требует ручного сбора информации из десятков источников».

## 2. Образ решения

**Научный клубок** — единая карта знаний (граф) для горно-металлургического R&D, которая связывает материалы, процессы, оборудование, эксперименты, публикации и экспертов в одну структуру, и отвечает на вопросы естественным языком со ссылками на источники.

**Чем это принципиально отличается от «просто RAG»:**
- Не «похожие куски текста», а **типизированный граф знаний** (13 типов сущностей, 17 типов связей) — можно пройти по цепочке материал→процесс→оборудование→результат, а не просто получить релевантный абзац.
- Каждый факт несёт **провенанс**: из какого документа взят (узел `Source`), какая у него достоверность (`confidence: high/medium/low`), когда он был извлечён (`extracted_at`) — отдельно от года публикации источника (`year`). Это прямая реализация требования ТЗ «модель верификации знаний: источник, уровень достоверности, дата актуализации».
- Числа **механически проверяются** против исходного текста документа — если модель извлечения "додумала" правдоподобное число, которого в тексте нет, факт помечается как неподтверждённый и не идёт в граф молча.
- Система умеет сказать «данных недостаточно» вместо того, чтобы выдумать ответ (см. пункт про `gaps` и `confidence: low` ниже) — важно для доверия в R&D-контексте, где ошибка в извлечённой концентрации или температуре недопустима.

## 3. Архитектура (для диаграммы)

```
Документ (PDF/DOCX, RU/EN)
   │
   ▼
[ML-1: Ingestion]  parse → chunk → LLM-извлечение (JSON Schema) → нормализация → верификация чисел
   │                                                                          │
   ▼                                                                          ▼
 Neo4j (граф: 13 типов узлов, провенанс)                        Qdrant (векторные эмбеддинги чанков)
   ▲                                                                          ▲
   │                                                                          │
[ML-2: Query]  NL-вопрос → QuerySpec → гибридный поиск (Cypher + вектор) → cross-encoder реранк → LLM-синтез
   │
   ▼
Backend (FastAPI, DDD/VSA) — auth/RBAC, оркестрация запроса, загрузка документов, дашборд
   │
   ▼
Frontend (React/TS) — поиск, ответ с цитатами, подграф (Cytoscape), логин, история, загрузка
```

Всё поднимается одной командой `docker compose up` (Neo4j + Qdrant + Postgres + Redis + MinIO + backend + worker + frontend). Асинхронная обработка документов — через arq/Redis-очередь, не блокирует API.

## 4. Онтология и охват корпуса (конкретные цифры для слайда)

- **13 типов сущностей**: Material, Process, Equipment, Property, Measurement, Condition, Experiment, Publication, Expert, Facility, Finding, Topic, Source.
- **17 типов связей**: uses_material, applies_to, operates_at_condition, has_measurement, measures_property, uses_equipment, produces_output, showed, described_in, authored_by, expert_in, conducted_at, validated_by, contradicts, supports, tagged, has_source.
- На момент сдачи в граф загружено **33 документа** (доклады, статьи, отчёты предоставленного корпуса, RU/EN) — **7200+ узлов**, автоматически через `POST /documents`, без единого ручного шага после загрузки файла.

## 5. Роли команды и что конкретно сделано

### ML-1 — Ingestion (извлечение знаний)

**Пайплайн**: parse (PyMuPDF для PDF, python-docx для DOCX, включая таблицы) → chunk (собственный recursive-splitter, без `langchain_text_splitters` — та библиотека сегфолтит на используемой версии Python) → LLM-извлечение по строгой JSON Schema (server-side constrained decoding — модель физически не может вернуть лейбл вне онтологии) → нормализация → запись в Neo4j с провенансом.

Конкретные задачи и решения:
- **Мультиязычность**: определение языка на уровне чанка (`langdetect`), сопоставление RU/EN синонимов через таблицу канонизации (`electrowinning`/`электроэкстракция`, `ПВП`/`печь взвешенной плавки`/`fluidized bed furnace` — ровно примеры из ТЗ) — одна и та же сущность не расползается на дубликаты только из-за разного языка упоминания.
- **Нормализация единиц измерения** через `pint`: концентрации приводятся к единому мг/л вне зависимости от того, как было в тексте (мг/дм³, г/л и т.д.).
- **Защита от галлюцинаций**: механическая сверка каждого извлечённого числа с сырым текстом документа. На реальном корпусе поймала конкретный случай — модель "восстановила" правдоподобное число (101325 Па — стандартное атмосферное давление) там, где в тексте была лишь качественная формулировка без числа. Непроверенные значения помечаются `verified: false`, а не тихо попадают в граф.
- **Версионирование фактов**: при повторном импорте, если значение измерения или confidence вывода реально изменились, старое состояние архивируется на отдельный узел `:Revision` со ссылкой на то, каким документом оно было заменено — прямая реализация пункта ТЗ «отслеживание изменений в выводах при появлении новых источников», которого не было бы при обычном `MERGE` (last-write-wins).
- **Провенанс на уровне факта**: каждый значимый вывод/измерение связан с узлом `Source` (конкретный фрагмент/страница документа), а не просто «откуда-то из документа X».

### ML-2 — Query & Retrieval (поиск и синтез ответов)

**Пайплайн**: вопрос на естественном языке → LLM парсит его в структурированный `QuerySpec` (материалы/процессы/условия/география/числовые ограничения/тип сравнения) → гибридный поиск (Cypher-обход графа + векторный поиск в Qdrant) → cross-encoder реранк объединённых кандидатов по реальной релевантности к вопросу → LLM-синтез структурированного ответа.

Конкретные задачи и решения:
- **Числовые диапазоны и условия** («концентрация сульфатов <200 мг/л» — дословный пример из ТЗ): нашёл и исправил 2 независимых бага, из-за которых такие запросы либо падали с ошибкой валидации, либо Cypher-фильтр никогда ни с чем не совпадал (сравнивал имя параметра не с тем узлом графа). Проверено синтетическим тестом до/после фикса.
- **Сравнительные запросы** («отечественная практика» vs «мировая практика», «вариант А» vs «вариант Б»): раньше сравнение парсилось, но не использовалось при поиске — ретрив шёл один раз, без разделения на стороны. Реализовал раздельный поиск по каждой стороне сравнения (по географии либо по конкретной сущности) с явной маркировкой находок, плюс **таблицы сравнения** в ответе, которые LLM заполняет только по тем критериям, что реально есть в найденных фактах — без выдумывания строк.
- **Cross-encoder реранкинг вместо RRF**: предыдущий метод объединения графового и векторного поиска (Reciprocal Rank Fusion) на практике оказался сломан — терял все метаданные и часть результатов. Заменил на конкатенацию с дедупликацией + реальный cross-encoder, который ранжирует кандидатов по семантической релевантности к тексту вопроса, а не просто по позиции в двух разных списках.
- **Синтез ответа**: структурированный JSON с cited-источниками (title, год, география, confidence, дата актуализации извлечения), явным указанием консенсуса и противоречий между источниками, честным `gaps` вместо выдумывания при нехватке данных, рекомендациями (похожие кейсы из смежных областей, смежные темы, эксперты/лаборатории).
- **Дашборд-аналитика для руководителя**: покрытие знаний по доменам (гидрометаллургия/металлургия/экология и т.д.), gap-detection (какие комбинации материал×процесс×условие не изучены — прямая реализация примера из ТЗ), зоны риска (темы с одним источником или с противоречиями).
- **End-to-end подключение**: раньше загрузка документа через UI ничего не делала с точки зрения извлечения (обработчик был заглушкой) — подключил реальный ML-1-пайплайн внутрь фонового воркера, с реальным падением/восстановлением при сбоях (arq иногда убивает долгую задачу по таймауту — обработал это так, чтобы документ не зависал вечно, а корректно помечался).

### Backend — 2 фулстек-разработчика

- FastAPI на DDD/Vertical-Slice-архитектуре: чёткое разделение по фичам (`features/auth`, `features/document`, `features/query`, `features/dashboard`), Dishka для DI.
- Аутентификация и ролевая модель (JWT, argon2), сид пользователей по ролям из ТЗ (исследователь, аналитик, руководитель, администратор, внешний партнёр).
- Загрузка документов (MinIO как объектное хранилище) → постановка в очередь фоновой обработки (arq/Redis) → статус-трекинг (`pending → processing → processed/failed`), эндпоинт списка документов.
- Оркестратор `/query/ask` — связывает парсер вопроса, графовый и векторный поиск, реранкер и синтез в единый HTTP-эндпоинт.
- Эндпоинт дашборда, CI/CD (GitHub Actions), инфраструктура (docker-compose для локальной разработки и деплоя, конфигурация reverse-proxy).

### Frontend — 1 разработчик

- React/TS-приложение (Vite, FSD-подобная структура: entities/pages/shared).
- Экран поиска с многоуровневой фильтрацией (материал/процесс/география/год/уровень достоверности).
- Экран ответа: текст + карточки источников (год, гео, достоверность, ссылка на фрагмент) + визуализация подграфа знаний (Cytoscape) — цепочка материал→процесс→оборудование→результат.
- Логин/сессия, история запросов, экран загрузки документов со статусами обработки в реальном времени.
